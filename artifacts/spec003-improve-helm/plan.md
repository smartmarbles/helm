# Implementation Plan: Improve Helm

> Generated by SAGE. Standard-Path, iterative plan for the vague request "improve Helm". Update progress as phases complete.

## Summary

Convert the open-ended "improve Helm" request into a repeatable audit→rank→ship→measure loop. Phase 1 produces a workspace health audit and ranked candidate list (no code changes). Phase 2 ships the first proof-of-value (PoV): one Small-effort improvement from the top-3, with an explicit before/after metric and acceptance test. Phases 3+ are added on-demand — one per improvement — keeping each iteration bounded and reversible. Key constraint: PoV must remain Small-scope and reversible; anything larger is deferred to a follow-up phase.

## Phases

### Phase 1: Audit & Rank — Discover what to improve

Goal: produce a signed audit with ranked candidates. Read-only phase; no production files edited.

- [ ] **Task 1.1**: Inventory agents, skills, templates, specs, and memory conventions; list observed issues and gaps with concrete evidence (file paths, line counts, broken refs). → **SCOOP** (research)
  - Files (read-only): `.github/agents/**`, `.github/skills/**`, `.github/templates/**`, `AGENTS.md`, `README.md`, `artifacts/**`, `synap.config.json`, `synap.lock.json`
  - Output: findings delivered to SAGE
  - Estimate: **Medium**
- [ ] **Task 1.2**: Run the existing skill validator and capture baseline output (pass/fail per skill, errors, warnings). → **PROBE** (evaluation)
  - Files (read/run): `artifacts/docs/validate_skill.py`, `.github/skills/**`
  - Output: `artifacts/spec003-improve-helm/validator-baseline.md`
  - Estimate: **Small**
- [ ] **Task 1.3**: Draft `audit.md`: merge SCOOP findings + PROBE baseline into ranked candidate table (≥5 rows) with Impact/Effort/Evidence/Owner; mark top-3 and the PoV pick. → **SAGE** (synthesis) with **QUILL** editorial pass
  - Files: `artifacts/spec003-improve-helm/audit.md`
  - Depends on: Task 1.1, Task 1.2
  - Estimate: **Medium**
- [ ] **Task 1.4**: Human checkpoint — user confirms PoV candidate pick before Phase 2 starts. → **ARTHUR** (gate)
  - Depends on: Task 1.3
  - Estimate: **Small**

> PARALLEL: Tasks 1.1 and 1.2 run simultaneously (no file overlap; SCOOP reads docs, PROBE runs validator).
> SEQUENTIAL: Task 1.3 blocked by 1.1+1.2; Task 1.4 blocked by 1.3.

**Phase 1 acceptance tests:**
- `audit.md` exists and covers all six areas (agents, skills, templates, specs, validator, risks).
- Candidate table has ≥5 rows with every column populated.
- Top-3 marked; exactly one flagged as PoV pick.
- `validator-baseline.md` exists with current pass/fail counts.

### Phase 2: Proof-of-Value — Ship the first improvement

Goal: implement exactly **one** Small-effort candidate from Phase 1 top-3, with a named before/after metric.

- [ ] **Task 2.1**: Scope-check the chosen PoV candidate; abort and re-pick if it no longer fits Small. → **SAGE**
  - Files: `artifacts/spec003-improve-helm/plan.md` (update this Phase 2 with chosen candidate ID)
  - Depends on: Task 1.4
  - Estimate: **Small**
- [ ] **Task 2.2**: Capture the "before" metric for the chosen candidate (line count, link failure count, validator error count, or doc-coverage %). → **PROBE**
  - Files: `artifacts/spec003-improve-helm/pov-metrics.md` (create)
  - Depends on: Task 2.1
  - Estimate: **Small**
- [ ] **Task 2.3**: Implement the improvement. → **implementer** (role depends on candidate; e.g., QUILL for doc/agent edits, PRISM for validator/script edits, MERLIN for roster/team changes)
  - Files: *determined by candidate* — recorded in Task 2.1
  - Depends on: Task 2.2
  - Estimate: **Small** (hard cap; if it grows, abort and open a new phase)
- [ ] **Task 2.4**: Re-capture the "after" metric; append before/after delta and retro note to `audit.md`. → **PROBE** + **QUILL**
  - Files: `artifacts/spec003-improve-helm/pov-metrics.md`, `artifacts/spec003-improve-helm/audit.md`
  - Depends on: Task 2.3
  - Estimate: **Small**
- [ ] **Task 2.5**: Run PoV acceptance test (see below). → **PROBE**
  - Depends on: Task 2.4
  - Estimate: **Small**

> SEQUENTIAL: All Phase 2 tasks are strictly sequential.
> BLOCKED BY: Phase 1 (all tasks).

**Phase 2 acceptance tests (explicit):**
1. `pov-metrics.md` contains a `before` and `after` value for the declared metric, and the delta is in the intended direction (e.g., errors decreased, coverage increased).
2. If the candidate touches `.github/skills/**`, re-running `artifacts/docs/validate_skill.py` shows no regressions vs. `validator-baseline.md`.
3. If the candidate touches an agent definition, PROBE runs a 1-prompt invocation smoke test against that agent and records pass/fail in `pov-metrics.md`.
4. `git diff` for the PoV change is reversible in a single revert commit (no unrelated edits mixed in).
5. A retro entry is appended to `audit.md` within the same iteration.

### Phase 3+: Iterative Improvements — One candidate per phase

Template for each post-PoV iteration. Add a new numbered phase (3, 4, 5, …) for each candidate executed.

- [ ] **Task N.1**: Select next candidate from ranked list; confirm still Small–Medium effort. → **SAGE**
- [ ] **Task N.2**: Capture before metric. → **PROBE**
- [ ] **Task N.3**: Implement. → **implementer** (role per candidate)
- [ ] **Task N.4**: Capture after metric; update `audit.md` retro. → **PROBE** + **QUILL**
- [ ] **Task N.5**: Run candidate-specific acceptance test. → **PROBE**

> SEQUENTIAL within each phase. Phases 3, 4, 5 are themselves sequential (one improvement at a time) unless explicitly flagged non-overlapping.

**Per-phase acceptance test:** candidate-specific metric improves in the stated direction; validator (if applicable) shows no regression; change is revertable in one commit.

### Final Phase: Close-out — Promote lessons to repo memory

- [ ] **Task F.1**: Summarize durable lessons (conventions discovered, recurring traps, useful commands) into `/memories/repo/helm-improvement-notes.md`. → **SAGE**
- [ ] **Task F.2**: Mark spec003 complete in `plan.md` Progress table; clear session checkpoints for this spec. → **ARTHUR**

> BLOCKED BY: All executed improvement phases.

## Recommended First-Phase PoV (minimal)

**Phase 1 is the PoV gate.** The minimal proof-of-value deliverable before any code change is:

1. `audit.md` with ranked candidate table (≥5 rows).
2. `validator-baseline.md` with current validator pass/fail counts.
3. One candidate explicitly marked **"PoV pick — Small effort, reversible"**.

These three artifacts are sufficient to demonstrate the loop works and to justify Phase 2 execution. If the user approves, Phase 2 ships the first real change under the Phase 2 acceptance tests above.

## Watch Out

- **Overlap with spec002**: `spec002-agent-system-hardening` is actively editing agents, skills, validator, AGENTS.md, and roster. Any spec003 candidate touching those files must be deferred or coordinated via ARTHUR to avoid write-contention. SCOOP must flag overlaps in Task 1.1.
- **Scope creep on PoV**: "While I'm in there, let me also…" is the failure mode. Task 2.3 is hard-capped at Small; if it grows, abort and open a new phase — do not silently expand.
- **Vague metrics**: "Feels better" is not a metric. Every candidate must declare a countable before/after value in Task 2.2.
- **Validator regressions**: If PoV touches skills, validator must be re-run; do not trust the absence of errors without comparing to `validator-baseline.md`.
- **Session checkpointing**: This is an iterative spec. After each phase (and after each major Phase 1 task), write a checkpoint to `/memories/session/sage-spec003-progress.md`. Clear it at close-out.
- **Candidate staleness**: Rankings may shift as spec002 lands changes. Re-score remaining candidates at the start of each new phase if spec002 has shipped work since the last scoring.
- **Implementer identity**: "implementer" is a role, not an agent. Every task assigning "implementer" must be resolved to a concrete agent (existing or MERLIN-hired temp) in Task 2.1 / N.1.
