# Feature Specification: Improve Helm

> Generated by SAGE from the open-ended request "improve Helm". This spec scopes the work as a **discovery-first, iterative improvement cycle** rather than a single feature. Update as findings land.

## Overview

"Improve Helm" is intentionally broad. This spec converts that into a repeatable improvement cycle: audit the current state of the Helm AI-team workspace, rank candidate improvements by impact and effort, then execute the top-ranked items in short, verifiable iterations. The first iteration delivers a minimal proof-of-value: a written health audit plus one shipped, measurable improvement.

## User Scenarios

### P1 — Evidence-based improvement loop

**As the** workspace owner, **I want** improvements to Helm grounded in a visible audit of the current state, **so that** each change targets a real gap instead of a guess.

**Acceptance Criteria:**
- [ ] A health audit document exists that inventories agents, skills, templates, specs, and known issues.
- [ ] The audit ranks at least 5 candidate improvements by impact (H/M/L) and effort (S/M/L).
- [ ] Each subsequent improvement iteration cites a ranked candidate as its justification.

### P2 — Proof-of-value first improvement

**As the** workspace owner, **I want** the first iteration to ship one small, measurable improvement, **so that** the cycle produces value before any large investment.

**Acceptance Criteria:**
- [ ] One improvement from the top-3 ranked candidates is implemented in Phase 2.
- [ ] The improvement has an explicit before/after metric (e.g., line count, broken link count, validator pass rate, doc coverage %).
- [ ] A short retrospective note is appended to the audit after the PoV ships.

### P3 — Iterative execution beyond PoV

**As the** workspace owner, **I want** further improvements to execute one at a time with the same audit→plan→ship→measure rhythm, **so that** scope stays bounded and progress stays visible.

**Acceptance Criteria:**
- [ ] After PoV, each additional improvement is tracked as a sub-phase with its own acceptance test.
- [ ] The audit is re-scored (or extended) when a major change lands.

## Requirements

### Functional Requirements

| ID | Requirement | Priority | Scenario |
|----|-------------|----------|----------|
| FR-001 | Produce a written `audit.md` covering: agents, skills, templates, existing specs, validator state, known risks. | Must | P1 |
| FR-002 | Rank candidate improvements in a table with columns: ID, Area, Impact (H/M/L), Effort (S/M/L), Evidence, Recommended owner. | Must | P1 |
| FR-003 | Identify the top-3 candidates and flag which is selected for the PoV iteration. | Must | P1 |
| FR-004 | Implement exactly one ranked improvement in the PoV iteration with a named before/after metric. | Must | P2 |
| FR-005 | Validate the PoV improvement with an explicit acceptance test (doc check, validator run, link check, or agent-invocation smoke). | Must | P2 |
| FR-006 | Record a retro entry (what changed, metric delta, lessons) in `audit.md` after PoV. | Must | P2 |
| FR-007 | For each post-PoV improvement, add a phase to `plan.md` with its own acceptance test before work begins. | Should | P3 |
| FR-008 | Preserve repo conventions: artifact location under `artifacts/spec###-*/`, session checkpoints under `/memories/session/`, repo memory under `/memories/repo/`. | Must | P1–P3 |
| FR-009 | No production code is edited during Phase 1 (audit is read-only). | Must | P1 |
| FR-010 | Cap PoV change size: Small effort, single-area scope, reversible. | Must | P2 |

### Key Entities

| Entity | Description | Key Attributes |
|--------|-------------|----------------|
| Audit | Snapshot of Helm workspace health. | areas[], risks[], candidates[] |
| Candidate | A proposed improvement. | id, area, impact, effort, evidence, owner |
| Iteration | One executed improvement. | candidate_id, metric_before, metric_after, acceptance_test |

## Success Criteria

| ID | Criterion | Measurable? |
|----|-----------|-------------|
| SC-001 | `audit.md` exists and covers all six areas in FR-001. | Yes |
| SC-002 | Candidate table has ≥5 rows, all columns populated. | Yes |
| SC-003 | PoV iteration ships with documented before/after metric. | Yes |
| SC-004 | PoV acceptance test passes (recorded in `plan.md` Progress table). | Yes |
| SC-005 | No spec folder naming, template, or memory-protocol convention is violated. | Yes |

## Edge Cases

- Audit surfaces zero high-impact issues → recommend closing the spec and logging the finding to `/memories/repo/`.
- PoV candidate turns out larger than Small effort once scoped → abort PoV, pick the next ranked candidate, do not silently expand scope.
- Two candidates tie on impact/effort → prefer the one with a simpler acceptance test.
- Overlap with active `spec002-agent-system-hardening` → defer overlapping candidates; note the deferral in the audit.

## Non-Functional Requirements

| Area | Requirement |
|------|-------------|
| Reversibility | PoV change must be revertable in a single commit. |
| Traceability | Every improvement cites a candidate ID from the audit. |
| Session Resumption | Checkpoints written to `/memories/session/` after each major artifact lands (audit, plan, PoV). |
