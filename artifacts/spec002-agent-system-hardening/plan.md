# Implementation Plan: Agent System Hardening

> Generated by SAGE from `spec.md` v2 (approved 2026-04-18). Phase numbering mirrors the spec's FR groupings (P1 → P11). Update as execution progresses.

## Summary

Harden the multi-agent orchestration system for reliable operation on GPT-4.1 without regressing the premium (Claude Sonnet/Opus) experience. Work is additive and measurement-gated: PROBE captures a GPT-4.1 baseline before any changes ship (Phase 1); memory model, skills, agent trims, and instructions rewrites land in the middle phases; PROBE re-runs at the end (Phase 11) to prove measurable improvement with no critical regressions. The key constraint is **no-code-implementer-agent on the team** — almost every deliverable is prose (agent prompts, instructions files, roster docs) or empirical test runs; the two code artifacts (`validate_skill.py` updates, `.gitignore` edit) require either a newly-hired temp coder or user implementation — flagged as Risk R-01.

## Technical Context

| Attribute | Value |
|-----------|-------|
| Primary artifacts | `.github/agents/*.agent.md`, `.github/skills/**`, `.github/scripts/validate_skill.py`, `copilot-instructions.md`, `AGENTS.md`, `.github/skills-roster.md` |
| Target behavior | VS Code Copilot agent orchestration (v1.105+) on GPT-4.1 and Claude Sonnet/Opus |
| Measurement | PROBE scorecard: 1–100 weighted overall + per-category sub-scores, model-tagged |
| Testing strategy | Empirical PROBE runs on GPT-4.1 (baseline + post-hardening delta); `validate_skill.py` on every skill; manual smoke tests for memory fallback and temp-discovery |

## Execution Conventions

- **ARTHUR dispatches one agent per task.** Independent tasks within a phase may be dispatched in parallel (explicit `> PARALLEL` markers below).
- **Checkpoint expectation**: each dispatched agent writes `/memories/session/<agent>-spec002-<slug>.md` after each major task completion, per FR-022.
- **Phase gate**: ARTHUR does not advance to the next phase until all tasks in the current phase have acceptance criteria met, unless a `> PARALLEL ACROSS PHASES` marker explicitly permits overlap.
- **Human checkpoints** (marked 🛑): ARTHUR pauses for user approval before proceeding past the gate.
- **Plan bookkeeping**: ARTHUR updates each task's checkbox to `[x]` and appends `*(complete YYYY-MM-DD)*` to the task line **immediately upon task completion** — before dispatching the next task. Dispatched agents must also confirm completion in their report so ARTHUR has a reliable trigger. This protects against plan drift across session boundaries (e.g., forced chat restarts after hiring a new agent).

## Critical Path

Phase 1 → Phase 2 → Phase 9a (validator relocation+updates) → Phase 3 (skill extraction) → Phase 4 (agent trims) → Phase 5 (copilot-instructions) → Phase 11 (final PROBE).
Approximate lower bound: 7 sequential phase-stages. Phases 6, 7, 8, 9b (roster + skill.instructions.md), and 10 can slot in parallel to the critical path.

## Parallelization Highlights

- **Phase 2 // Phase 6 // Phase 7 // Phase 9a** — all run after Phase 1 gate; touch disjoint files.
- **Phase 3 skill authoring** — per-agent skills run in parallel (ARTHUR first as template, then remaining five in parallel).
- **Phase 4 per-agent trims** — run in parallel (one agent file each).
- **Phase 8 // Phase 10** — both are documentation-heavy, no file overlap; dispatch concurrently.
