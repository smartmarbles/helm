# Feature Specification: Agent System Hardening

> Generated by SAGE. This spec is a workspace artifact — update it as requirements evolve.

## Overview

Harden the Helm multi-agent orchestration system so it performs consistently on weaker models (specifically GPT-4.1) without degrading the premium-model (Claude Sonnet/Opus) experience. Work is additive: agent files shrink and move process detail into skills, `copilot-instructions.md` becomes self-sufficient with explicit tool bans and worked examples, the memory model gains graceful degradation, the temp-agent lifecycle gets verified, and a PROBE-measured baseline on GPT-4.1 gates before/after comparison.

**Context note on research inputs:** This spec was authored from the task brief's summaries of two SCOOP research outputs (`scoop-openclaw-orchestration.md`, `scoop-vscode-memory-api.md`). The session memory tool was unavailable during spec authoring — a real-world instance of the failure mode Phase 2 addresses. (Resolved mid-spec: frontmatter entry `vscode/memory` has since been added to SAGE so subsequent revisions can read prior checkpoints directly.) Findings relied upon: subagents default to no memory tool (intentional v1.105 isolation; opt-in via frontmatter), OpenClaw's adoptable patterns (runtime tool filters — likely infeasible here, "NOT for:" skill clauses, prompt-cache stability via deterministic ordering, char-budgeted injection).

## User Scenarios

### P1 — GPT-4.1 fallback session stays reliable

**As a** user whose premium Copilot request budget is exhausted and has fallen back to GPT-4.1, **I want** the orchestration system to continue following its delegation protocol, respect tool restrictions, and produce correctly-scoped artifacts, **so that** I can keep working productively without rewriting outputs or babysitting agents.

**Acceptance Criteria:**
- [x] ARTHUR delegates output-producing work to agents rather than writing artifacts directly, on GPT-4.1, in ≥90% of PROBE test cases
- [x] Agents with forbidden tools (e.g., ARTHUR with `create_file`) do not invoke those tools in PROBE test runs on GPT-4.1
- [x] PROBE overall adherence score on GPT-4.1 improves by a measurable delta vs. the Phase 1 baseline, with no critical regressions
- [x] The "write me a README" / trivial-seeming delegation case routes through `runSubagent(QUILL)` rather than direct file creation on GPT-4.1

### P2 — Agent sessions degrade gracefully when memory is unavailable

**As a** user running a subagent that does not have the memory tool granted (or a session where the memory tool is disabled), **I want** the agent to detect the absence, fall back to a workspace-local checkpoint location, and notify me once per session, **so that** long-running work is not lost and I know about the degraded mode.

**Acceptance Criteria:**
- [x] Agents probe memory availability at startup via `view /memories/session/`
- [x] Agents without memory write checkpoints to `.agent-memory/{user,session,repo}/` (gitignored)
- [x] Agents prepend `[no-memory]` to their final reply exactly once per session when degraded
- [x] The fallback behaviour is documented in `/memories/repo/` (or `.agent-memory/repo/` in degraded mode) so future sessions discover it
- [x] Subagents that DO have memory access (SAGE, SCOOP, PROBE) use per-agent checkpoint filenames `<agent>-<slug>.md` and do not collide

### P3 — Skills and temp agents are visible and trustworthy

**As a** user onboarding a new skill or temp agent, **I want** a validator that enforces the Copilot-native skill contract and a roster that shows each skill's validation status and warnings, **so that** reliability risks are visible before the skill ships.

**Acceptance Criteria:**
- [x] `validate_skill.py` runs from `.github/scripts/` and enforces the customized contract (evals required, "NOT for:" clause checked, frontmatter allowlist widened for Copilot, etc.)
- [x] `.github/skills-roster.md` lists every skill with purpose, used-by, validation date, warnings
- [x] Temp agents are verified not to leak into Copilot's agent discovery context when archived
- [x] MERLIN's hiring flow produces a draft skill + validator run + minimum-viable eval pass alongside the agent file

## Edge Cases

- **Memory tool flickers mid-session** — agent probed successfully at startup but a later call fails. Degrade to fallback mid-session and emit `[no-memory]` at next reply; do not re-probe every call.
- **Concurrent subagents write to the same checkpoint slug** — last-writer-wins per SCOOP research; per-agent `<agent>-<slug>.md` naming prevents collisions between different agents but not between parallel instances of the same agent. Document the limitation; do not attempt locking.
- **Cross-workspace user memory leakage** — notes written to `/memories/` are loaded into every Copilot session in every workspace (first 200 lines). Documented in the new `AGENTS.md` Memory Scope Convention: agents default to writing project-specific durable knowledge to `/memories/repo/`, reserving `/memories/` (user scope) for content that genuinely crosses projects. No config file or namespace subfolder required — the extension routes by path prefix automatically.
- **GPT-4.1 invokes forbidden tools despite prose instructions** — expected residual failure mode; tracked via PROBE tool-restriction adherence score. Not a blocker unless critical regression vs baseline.
- **Temp agent auto-discovery** — if Copilot recurses `.github/agents/temps/`, archived temps silently inflate every session's context. Phase 7 empirical test settles this before choosing the archive location.
- **Validator false positives on Copilot-native frontmatter fields** — current allowlist may be too narrow. Phase 9 audit widens it; regressions surface as roster warnings rather than hard failures.
- **Skill eval prompts drift from skill content** — evals/evals.json exists but doesn't exercise what the skill actually does. Minimum-viable pass catches gross mismatches; full drift detection is out of scope.
- **User manually edits a skill bypassing MERLIN** — validator still runs on demand; roster update via `--update-roster` keeps the index accurate. Documented in `.github/skills/skill.instructions.md`.

## Key Decisions (recorded for future reference)

1. **Roleplay-default ARTHUR, not always-dispatch** — continuous context is worth preserving for premium UX. Tool-restriction via prose is accepted as the tradeoff on weaker models; PROBE measures the cost.
2. **Subagents default to NO memory tool; opt-in per agent** — aligns with Microsoft's intentional v1.105 isolation design. SAGE, SCOOP, PROBE opt in; QUILL, MERLIN, temps stay out.
3. **Memory scope discipline over namespace config** — no `helm.config.json`, no `/memories/helm-team/` subfolder. Agents choose the correct scope (user / repo / session) based on content portability, documented in `AGENTS.md`. Path prefixes route automatically.
4. **No CI / pre-commit hooks** — user declined the overhead. Validator runs on demand; skills roster surfaces reliability risk instead.
5. **No trivial/quick routing path** — GPT-4.1 empirically abuses it as a planning escape. Routing table has exactly three paths: research / full / standard, with explicit three-step classification protocol (trigger phrases → semantic match → default to standard).
6. **Skills roster in lieu of CI** for reliability visibility.
7. **Validator is a universal asset**, not agent-specific, located at `.github/scripts/validate_skill.py`.
8. **VS Code Copilot-native skill format** — not portable to agentskills.io. Vendor-term check dropped, frontmatter allowlist widened for Copilot-native fields. Quality rules (triggerable descriptions, body size, kebab-case names, dual-mode scripts, eval coverage) retained.
9. **Skill contract lives in a Copilot instructions file** (`.github/skills/skill.instructions.md` with `applyTo: ".github/skills/**"`) so it activates only in skill-authoring context rather than polluting the base prompt.
10. **MERLIN uses `skill-creator` as automatic reference during hiring; interactive iteration is user-invoked** — two distinct modes documented to prevent conflation.
11. **Agent prompts are portable** — no Helm-specific language in agent definitions or shared docs. File paths, workspace structure, and artifact conventions stay universal.
12. **Distribution/portability work is deferred** to spec003 (see Deferred section).

## Deferred to spec003 — Distribution & Portability

The following concerns are explicitly **out of scope for spec002** and will be addressed in a follow-on spec:

- **Installation / adoption flow** — how a new project pulls the agent framework into its `.github/` without manual copy.
- **Versioning & upgrade path** — how downstream projects track framework updates and migrate when contracts change.
- **Framework-vs-project separation** — identifying which files are framework assets (owned upstream) vs project-specific customizations (owned locally) and how they coexist.
- **Multi-project sharing of skills and agents** — whether skills/agents can be pulled from a shared registry or remain vendored per project.

**Rationale for deferral:** spec002 focuses on hardening the orchestration system in-place for reliable GPT-4.1 operation. Portability design requires the hardened system as a stable baseline; attempting both at once risks scope creep and rework. Decision 5 (removing Helm-specific language from agent prompts) is the only portability-adjacent change retained in spec002 because it is trivially additive and unblocks spec003.
